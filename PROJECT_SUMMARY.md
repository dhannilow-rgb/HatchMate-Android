## 💡 Key Architectural Decisions

### 1. Closed-Loop Design
**Why**: Ensure data consistency & eliminate manual entry errors
**How**: @Transaction methods that atomically update multiple tables
**Benefit**: Single user action triggers multiple database operations reliably

### 2. Single Source of Truth (SSOT)
**Why**: Prevent pricing conflicts
**How**: Master Harga controls all transaction prices
**Benefit**: Change one master price, all future transactions use new price

### 3. StateFlow for Reactivity
**Why**: Auto-update UI when data changes
**How**: DAO returns Flow, ViewModel converts to StateFlow
**Benefit**: Whenever database changes, UI automatically re-renders

### 4. Room Database
**Why**: Type-safe, compile-time verified SQL
**How**: Entity classes + DAO interface
**Benefit**: Catch database errors at compile-time, not runtime

### 5. MVVM Architecture
**Why**: Clean separation of concerns
**How**: UI (Composable) → ViewModel → DAO → Room
**Benefit**: Easy to test, easy to maintain, easy to scale
